# Phase 999.27: Stepper Mode for Pipeline Steps - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a step-by-step execution mode to the existing pipeline so users can run each of the
four pipeline stages independently (extract audio → transcribe → translate → write SRT).
Each step saves its output as an intermediate artefact so subsequent steps can resume
without repeating prior work — solving the case where a late-stage failure (e.g. a
translation crash) discards all previously completed work.

The full "all steps in one shot" flow remains the default (one-click UX preserved).

All three surfaces are in scope: CLI, GUI, and FastAPI REST API.

</domain>

<decisions>
## Implementation Decisions

### Pipeline Stage Order
- **D-01:** Four stages: (1) Extract Audio, (2) Transcribe, (3) Translate, (4) Write SRT.
  This matches the current `run_pipeline()` stage sequence exactly.

### Intermediate Artifact Storage
- **D-02:** User-specified output directory — a single folder the user picks upfront before
  starting any step. All intermediate files for that job land in this one directory.
- **D-03:** Single output dir for all steps. The final SRT goes into this same directory
  (no separate --output separate from the work dir for step mode).
- **D-04:** Intermediate files are **kept until manually cleared** — they persist across
  app restarts so users can resume from any stage. A "Clear work" button (GUI) or
  equivalent CLI command removes them.

### Segments Serialization
- **D-05:** Transcription segments are serialized to **JSON** (human-readable, editable,
  not Python-version fragile).
- **D-06:** Schema: flat JSON array of objects — `[{start, end, text}, ...]`. One file for
  raw transcription output, one for translated segments (if translation ran).
- **D-07:** Filenames use stage names:
  - `audio.wav` — output of step 1 (Extract Audio)
  - `transcription.json` — output of step 2 (Transcribe)
  - `translation.json` — output of step 3 (Translate), only written when translation runs
  - `subtitles.srt` — output of step 4 (Write SRT)

### Translate Step When No Target Language
- **D-08:** The Translate step is **always shown** in the stepper widget. When no target
  language is configured it is marked as **Skipped** (visual "Skipped" badge, button
  disabled) rather than hidden.
- **D-09:** The user **can** set a target language after step 2 (Transcribe) completes
  and then run the Translate step retroactively. The step activates as soon as a target
  language is configured, using `transcription.json` as input.

### CLI Interface
- **D-10:** New flag `--step extract|transcribe|translate|write` — runs a single named
  stage and exits.
- **D-11:** New flag `--work-dir` — path to the directory where intermediate artefacts
  are read from / written to when using step mode.
- **D-12:** Existing `--input` / `--output` / `--target-lang` etc. flags remain unchanged
  for full-run mode. Step mode uses `--work-dir` + `--step` instead.

### GUI Interface
- **D-13:** A horizontal stepper widget below (or above) the existing controls shows the
  four stages with completed/current/pending visual states.
- **D-14:** Each stage has an action button that is **only active when its prerequisite
  output exists** in the work dir (e.g. "Transcribe" requires `audio.wav` to be present).
- **D-15:** A single "Generate subtitles (all steps)" button remains as the default action
  — existing one-click UX is preserved.
- **D-16:** A "Clear work" button removes all intermediate files from the work dir.

### FastAPI REST API
- **D-17:** Step mode is in scope for the API in this phase. Separate endpoints per stage:
  - `POST /extract` — upload video → returns WAV
  - `POST /transcribe` — upload WAV → returns `transcription.json`
  - `POST /translate` — upload `transcription.json` + target lang → returns `translation.json`
  - `POST /write` — upload segments JSON → returns `.srt` file
- **D-18:** API step mode is **stateless** — the client uploads the input artefact for
  each step and receives the output artefact in the response. No server-side session/job
  state is maintained. This matches the existing `POST /subtitles` stateless pattern.
- **D-19:** The existing `POST /subtitles` full-pipeline endpoint is unchanged.

### Relationship to Phase 999.26 (Console Log)
- **D-20:** Each step completion is a natural log event that should surface via the console
  log panel (Phase 999.26). The stepper and console log are complementary — this phase
  should emit step-level log messages that Phase 999.26 can display, without hard-coupling
  to it.

### Agent's Discretion
- Exact stepper widget component (CustomTkinter — CTkFrame, CTkSegmentedButton, or custom
  drawn widget) — planner chooses the most practical approach within CTk.
- Whether each step runs in the same thread pool executor as the current pipeline or gets
  its own async task model for the API layer.
- JSON schema version field (optional) — agent may add a `version` or `schema_version`
  field to segment JSON files if it reduces future fragility.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `run_pipeline()` in `pipeline.py` — already has 4 clearly delimited stages with
  `progress_callback("stage name", current, total)` calls. Decompose by extracting each
  stage block into its own callable.
- `audio_temp_context()` — currently used for temp WAV; in step mode the WAV is written to
  the work dir instead (not cleaned up automatically).
- `WhisperTranscriber.transcribe()` returns a `TranscriptionResult` with `.segments`,
  `.language`, and `.duration` — the segments list is what gets serialized to JSON.

### Established Patterns
- Lazy imports inside `run_pipeline()` (audio, srt_writer, transcriber, translator) to
  avoid import-time side effects — step-mode functions should follow the same pattern.
- `progress_callback: Optional[Callable[[str, int, int], None]]` — existing callback
  signature; step mode can fire callbacks per step naturally.
- FastAPI thread pool for CPU-bound work (`run_pipeline` dispatched via
  `asyncio.get_event_loop().run_in_executor(None, ...)` or similar) — step endpoints should
  use the same pattern.

### Integration Points
- GUI `GenSubtitlesApp` in `gui/main.py` is where the stepper widget and step buttons are
  added. The existing "Generate" button flow shows the pattern for invoking pipeline work
  in a background thread.
- CLI `generate()` callback in `cli/main.py` — new `--step` / `--work-dir` flags branch
  into step logic before the existing `run_pipeline()` call.
- API router `subtitles.py` — existing router file where new `/extract`, `/transcribe`,
  `/translate`, `/write` endpoints are added alongside existing `POST /subtitles`.

</code_context>

<specifics>
## Specific Ideas

- Work dir lifecycle: "kept until manually cleared" means the GUI Clear button and a CLI
  `--clear-work` sub-flag (or separate `gensubtitles clear-work --work-dir <path>`) that
  removes all intermediate files from the work dir.
- Flat JSON array `[{start, end, text}]` for both `transcription.json` and
  `translation.json` means the write step only needs the segments — it does not need to
  know which step produced them (same interface for translated and untranslated paths).
- Stepper guards (step button only active when prerequisite file exists) act as the
  validation layer — no special "state machine" needed beyond file-existence checks in
  the work dir.

</specifics>
