<decisions>
## Implementation Decisions

### D-01: README Structure
Reorganize both READMEs to an audience-split structure:

```
1. (Header / badges / one-liner)
2. Installation
3. Quick Start (pick your usage mode: GUI / CLI / API)
4. GUI Usage
   - Generating subtitles
   - Translation settings
   - Configuring the app (Settings dialog)
   - Help menu
5. CLI Usage
   - generate subcommand (all flags)
   - translate subcommand
   - convert subcommand
   - serve subcommand
6. API Usage
7. Configuration (settings.json location, schema)
8. Troubleshooting
9. License
```

The current flat structure (Installation → CLI → API → Translation → Troubleshooting) is replaced entirely. GUI gets equal billing alongside CLI.

### D-02: GUI Section Format
- Sub-sections per GUI area: main workflow, translation settings, Settings dialog, Help menu
- Inline in README.md (not a separate docs/gui.md)
- No screenshots — deferred to a future phase
- Each sub-section: step-by-step prose, ~10-20 lines

### D-03: Translation Engine Coverage
- Document Argos Translate fully (default offline engine, works in CLI + GUI)
- DeepL and LibreTranslate: one-liner each pointing to `--engine --help` for setup
- No status table — keep it simple
- Note: "DeepL and LibreTranslate are available via CLI (`--engine deepl`/`--engine libretranslate`); GUI support coming in a future release."

### D-04: README.es.md Sync Strategy
- README.es.md must match the section headings and structure of README.md exactly
- Spanish prose can be shorter/simpler where translation is hard
- Code examples, command snippets, flag tables: kept in English (same as README.md)
- No banner needed — both files are updated in this phase

### Agent's Discretion
- CLI flags table: agent decides whether to keep as a single table or split by subcommand
- Exact prose wording for each section
- Whether `convert` subcommand gets its own sub-section or is folded into CLI misc
- Settings JSON schema detail level (field names + defaults vs full description)

</decisions>
