# Phase 999.24: REFACTOR — Localisation Separation - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Move the `_STRINGS` dictionary and localisation lookup logic out of `GenSubtitlesApp`
in `gensubtitles/gui/main.py` into a new dedicated module `gensubtitles/gui/locale.py`.
Update all call sites in `main.py` to use the new module's public API.

Scoped to `gensubtitles/gui/` only. No API, pipeline, CLI, or settings changes.

**Pattern context:** Follows the refactor series — `theme.py` owns colors/typography,
`styles.py` owns spacing/dimensions, `locale.py` owns localisation strings and lookup.

</domain>

<decisions>
## Implementation Decisions

### New Module
- **D-01:** Create `gensubtitles/gui/locale.py` as the destination for all localisation
  concerns. Name chosen as "locale" (not "strings", "i18n") to align with standard
  Python/OS terminology for language-specific resources.

### What Moves
- **D-02:** Move the following from `gui/main.py` to `gui/locale.py`:
  - `_STRINGS` dictionary (the full en/es translation registry)
  - `set_language(lang: str) -> None` — new public function to set the active language
  - `s(key: str) -> str` — new public lookup function (replaces `self._s()`)
- **D-03:** `_relabel_tabview_headers()` and `_apply_ui_language()` **stay in `main.py`**
  — they call `widget.configure()` and are tightly coupled to widget references.
  They update to call `s()` (imported from `locale`) instead of `_STRINGS` directly.

### Public API Shape
- **D-04:** `locale.py` exposes:
  - `set_language(lang: str) -> None` — sets module-level active language (called by
    `main.py` at startup and when the user saves a UI language change in Settings).
  - `s(key: str) -> str` — returns the localized string for the active language.
    Falls back to English if key is missing in the target locale, then returns key
    as-is if missing from English too (same fallback behaviour as current `_s()`).
  - `_STRINGS` remains private (underscore-prefixed) inside `locale.py`.

### Language Resolution
- **D-05:** Module-level mutable state — `locale.py` holds a module-level `_active_lang`
  variable initialized to `"en"`. `set_language()` updates it; `s()` reads it.
  No settings import inside `locale.py` — the caller (`main.py`) is responsible for
  calling `set_language()` with the correct value read from settings.

### Call Site Migration
- **D-06:** `main.py` imports `from .locale import s, set_language`.
  - All `self._s(key)` calls become `s(key)`.
  - The `_s()` method on `GenSubtitlesApp` is removed entirely.
  - Any direct `_STRINGS[lang][key]` references in `main.py` are replaced with `s(key)`.
  - Any `_STRINGS["es"][key]` references (e.g., in `_relabel_tabview_headers`) use 
    `_STRINGS` imported from `locale` only where lang-specific lookup is needed —
    or refactor the callers to use `s()` with a temporary `set_language()` call if cleaner.

### Agent's Discretion
- Whether `_STRINGS` needs to be exported from `locale.py` at all (for use in
  `_relabel_tabview_headers` which currently accesses `_STRINGS["es"]` directly),
  or whether a helper like `s_lang(key, lang)` is added to avoid importing the private dict.
- Exact placement of `set_language()` call(s) in `main.py` — at least at app init
  (reading `ui_language` from settings) and after settings save.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `_STRINGS` dict already fully populated for `"en"` and `"es"` (~170 lines in `main.py`)
- `_s(self, key)` method already implements the correct fallback logic
- `_apply_ui_language()` already drives widget re-labeling via `_s()`

### Established Patterns
- `theme.py` and `styles.py` set the pattern: module-level private data, public accessor functions,
  imported at the top of `main.py` with `from .XXX import func`
- Apply-after-construction pattern in use for styles; `set_language()` follows a similar
  "configure once, use everywhere" approach

### Integration Points
- `main.py` `__init__` must call `set_language(settings.ui_language)` before any `s()` calls
- `_save_settings()` must call `set_language(new_lang)` after persisting settings
- `_relabel_tabview_headers()` currently accesses `_STRINGS["es"]` directly — needs
  a decision on whether to expose a `s_lang(key, lang)` helper or restructure

</code_context>

<specifics>
## Specific Ideas

- Module named `locale.py` (user's explicit choice over `strings.py` and `i18n.py`)
- `s(key)` as the lookup function name (consistent with `p()` and `font()` from `theme.py`)
- Module-level active language via `set_language()` — no settings coupling inside `locale.py`
