## Area: New module name

**Q:** What should the new module be called?

| Option | Description |
|--------|-------------|
| `strings.py` | Matches existing pattern: theme.py → colors/typography, styles.py → spacing/dimensions |
| **`locale.py`** ✓ | Standard Python i18n convention — aligns with OS/Python terminology for language-specific resources |
| `i18n.py` | Short alias — familiar but heavier connotation than what we're doing |

**Selected:** `locale.py`

---

## Area: Public API shape

**Q (1/2):** How should the lookup function know the active language?

| Option | Description |
|--------|-------------|
| **Module-level active language — `set_language()` + `s(key)`** ✓ | Module holds `set_language(lang)` + `s(key)`. Zero extra args at every call site. |
| Explicit lang per call — `s(key, lang)` | Pure but every call site must pass the lang |
| Locale object / dict injected at construction | More OO but adds an object to thread through the app |

**Selected:** Module-level active language — `set_language()` + `s(key)`

---

**Q (2/2):** What should the lookup function be named?

| Option | Description |
|--------|-------------|
| **`s(key: str) -> str`** ✓ | Matches `theme.py`'s `p()` and `styles.py` convention — short, public, consistent |
| `translate(key: str) -> str` | Explicit but verbose at 60+ call sites |
| Keep `_s()` as module-level | Underscore prefix — less consistent with the refactor pattern |

**Selected:** `s(key: str) -> str`

---

## Area: _apply_ui_language() scope

**Q:** Where should `_apply_ui_language()` (widget re-labeling logic) live?

| Option | Description |
|--------|-------------|
| **Widget updating stays in `main.py` — `locale.py` is pure data + lookup** ✓ | `_apply_ui_language()` stays in `main.py`; `locale.py` only owns the dict + `s()` + `set_language()` |
| Move widget updating to `locale.py` as `apply_ui_language(app)` | Would couple `locale.py` back to widget internals |
| `locale.py` stateless, `main.py` calls `s()` inside `_apply_ui_language()` | Equivalent to option 1 but with explicit stateless constraint |

**Selected:** Widget updating stays in `main.py` — `locale.py` is pure data + lookup

---

## Area: What _STRINGS-related code moves

**Q:** What exactly moves to `locale.py`?

| Option | Description |
|--------|-------------|
| **`_STRINGS` + `set_language()` + `s()` — full localisation data layer** ✓ | `_s()` method in `main.py` removed; call sites updated to import `s()` |
| Only `_STRINGS` dict moves | Lookup logic stays in `main.py` — partial refactor |
| Everything including `_relabel_tabview_headers()` | Too much coupling to widget state |

**Selected:** `_STRINGS` + `set_language()` + `s()` — full localisation data layer
