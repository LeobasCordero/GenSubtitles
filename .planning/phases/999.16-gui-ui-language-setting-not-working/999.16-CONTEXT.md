# Phase 999.16: GUI — UI Language Setting Not Working - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the UI Language setting in Settings → Preferences so that selecting a different language immediately relabels all visible text in the app. The setting is already persisted via AppSettings — it just never applies to any widget. Scope: fix the runtime language-switch mechanism only. No new languages, no new settings fields.

</domain>

<decisions>
## Implementation Decisions

### Switch Timing
- **D-01:** Language switch applies **immediately** on Save — no restart required. Consistent with how Appearance Mode already works in `_save_settings`.

### Implementation Approach
- **D-02:** **Relabeling pass** — keep existing widget tree, store refs for all labeled widgets, and call `.configure(text=...)` on each after Save with the new locale's strings. Do NOT destroy/rebuild the widget tree (full re-render is out of scope).

### Scope of Translatable Strings
- **D-03:** **Full scope** — translate all visible text:
  - Main UI: all `CTkLabel` text, button labels (`CTkButton`), tab names, option menu labels
  - Menu bar: cascade names, menu item labels (Settings, Preferences…, Help, Tutorial, Available Languages, About GenSubtitles)
  - Settings panel: field labels, Save/Back buttons, header
  - Dialog content: Tutorial window text, Available Languages dialog, About GenSubtitles dialog
  - Error messages surfaced via `messagebox` (these can use the live locale at call time — no pre-storage needed)

### String Registry Design
- **D-04:** Implement a minimal `_STRINGS` dict keyed by locale code (`"en"`, `"es"`) in `gui/main.py`. Each locale entry maps string keys to translated text. The agent picks the structure and key names — user has no preference on internals.
- **D-05:** A module-level helper `_s(key)` (or `_t(key)`) reads the current locale from `self._current_settings.ui_language` and returns the translated string. Called at relabeling time, not at widget creation time (widgets are built once with hardcoded English, then relabeled on language change).

### the agent's Discretion
- Exact string key names and dict structure
- Whether to define the registry as a module-level constant or a method
- How to handle missing keys (fallback to English)
- Whether to store translatable CTkLabel refs in a list or named attributes
- The exact Spanish translations for each UI string (use natural, idiomatic Spanish)

</decisions>

<specifics>
## Specific Ideas

- The pattern is already established for Appearance Mode: `_save_settings` calls `ctk.set_appearance_mode(new_settings.appearance_mode)` immediately. UI Language should follow the same inline-application pattern.
- `_build_menu_bar` uses `tk.Menu` which requires `menubar.entryconfigure(index, label=...)` for relabeling — distinct from CTk `.configure(text=...)`.
- Dialogs (`_show_tutorial`, `_show_about`, `_show_language_pairs`) are built fresh on each open — they can read the current locale at open time rather than needing relabeling.
- `_apply_startup_settings` loads `ui_language` on startup — the initial language can be applied right there by calling the relabel pass after `_build_ui`.

</specifics>
