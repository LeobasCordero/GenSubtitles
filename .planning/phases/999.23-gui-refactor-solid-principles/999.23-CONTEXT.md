# Phase 999.23: REFACTOR — Apply SOLID Principles to GUI - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Apply the **Single Responsibility Principle** to `gensubtitles/gui/main.py` by extracting
the server lifecycle concern into a new dedicated module `gensubtitles/gui/server.py`.

**Narrowed scope (from discussion):** SRP only. The other four SOLID principles
(OCP, LSP, ISP, DI) have no natural application points in a small single-window
desktop app and are explicitly deferred. This phase is purely about decomposing the
god-class by responsibility.

**This phase extracts the server lifecycle cluster only.** Decomposition of other
clusters (job execution, OS theme sync, help dialogs, settings persistence) is left
for future phases as the refactor series continues.

Scoped to `gensubtitles/gui/` only. No API, pipeline, CLI, or settings changes.

**Pattern context:** Continues the refactor series — `theme.py` (colors/typography) →
`styles.py` (spacing/dimensions) → `locale.py` (localisation strings) →
`server.py` (server lifecycle management).

</domain>

<decisions>
## Implementation Decisions

### Scope
- **D-01:** SRP only. OCP, LSP, ISP, and DI are deferred — not applicable to this
  codebase's single-class, single-window structure at this time.
- **D-02:** One decomposition target this phase: the **server lifecycle cluster**.
  Other clusters (job execution, help dialogs, OS theme sync, settings) stay in
  `GenSubtitlesApp` until future refactor phases.

### New Module
- **D-03:** Create `gensubtitles/gui/server.py` following the same standalone module
  pattern established by `theme.py`, `styles.py`, and `locale.py`. No new classes —
  module-level functions and state.

### What Moves to `server.py`
- **D-04:** The following move from `main.py` to `server.py`:
  - Module-level constants: `_SERVER_PORT`, `_BASE_URL` (currently in `main.py`)
  - `_start_server()` logic — uvicorn config, thread creation, the `_wait_for_server`
    inner polling loop
  - `_stop_server()` logic — `server.should_exit = True`
  - Module-level `_server` instance variable (currently `self._server` on the app)

### Callback-Based Decoupling
- **D-05:** `server.py` must remain **widget-free** (no imports from `ctk`, no reference
  to `GenSubtitlesApp`). Since the server startup process needs to update the GUI, it
  accepts three callbacks at start time:
  - `on_progress(message: str, progress: int) -> None` — called while server is booting
    (model downloading / loading); maps to `_apply_startup_progress`
  - `on_ready() -> None` — called when server is ready; maps to `_on_server_ready`
  - `on_failed(detail: str) -> None` — called on error; maps to `_on_server_failed`
  
  Callbacks are scheduled via `self.after(0, ...)` by the caller (`GenSubtitlesApp`) —
  NOT inside `server.py`. The server module only calls the callbacks directly; threading
  concerns of marshalling back to the Tkinter main thread remain the caller's
  responsibility.

### What Stays in `main.py`
- **D-06:** The following stay in `GenSubtitlesApp` (they touch widgets directly):
  - `_on_server_ready()` — enables buttons, hides progress bar, populates dropdowns
  - `_on_server_failed()` — shows error in progress bar / stage label
  - `_apply_startup_progress()` — updates progress bar mode and stage label
  - `on_closing()` — calls `server.stop()` instead of `self._stop_server()`

### Public API of `server.py`
- **D-07:** `server.py` exposes:
  - `start(on_progress, on_ready, on_failed, is_closing) -> None` — starts uvicorn thread
    and polling thread. `is_closing` is a callable returning `bool` (replaces reading
    `self._closing` directly — breaks the widget coupling).
  - `stop() -> None` — sets `server.should_exit = True` on the module-level `_server`.
  - `SERVER_PORT: int` and `BASE_URL: str` — public constants, used by `main.py` and
    potentially by other modules that need to know the server address.
  - `_server` (private) — the `uvicorn.Server` instance; module-level, not exposed.

### No Dependency Injection
- **D-08:** DI is explicitly deferred. `GenSubtitlesApp` continues to call `server.start()`
  directly. No constructor injection, no protocol/interface abstraction.

### the agent's Discretion
- Exact bind timeout value (`_SERVER_BIND_TIMEOUT = 30`) stays with the server module.
- Module docstring style should match `theme.py` / `styles.py` / `locale.py`.
- Whether `is_closing` is a `Callable[[], bool]` or a `threading.Event` — planner decides
  whichever is cleaner given the existing code.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `_SERVER_PORT = 8000` and `_BASE_URL = f"http://127.0.0.1:{_SERVER_PORT}"` — module-level
  constants in `main.py` today; will become `server.SERVER_PORT` and `server.BASE_URL`
- `uvicorn.Server` / `uvicorn.Config` — already a dependency in `requirements.txt`

### Established Patterns
- All prior refactor modules (theme, styles, locale) use `from __future__ import annotations`
  and expose only what's needed (no star exports)
- `main.py` uses `self.after(0, lambda: ...)` to marshal callbacks to the Tkinter thread —
  this pattern must be preserved; `server.py` should NOT call `self.after()`

### Integration Points
- `GenSubtitlesApp.__init__` calls `self._start_server()` at the end — will become
  `server.start(on_progress=..., on_ready=..., on_failed=..., is_closing=...)`
- `on_closing()` calls `self._stop_server()` — will become `server.stop()`
- `_run_sse_flow` and `_run_translate` use `_BASE_URL` — will import from `server`

</code_context>

<specifics>
## Specific Ideas

- Same extraction pattern as locale.py: move code, expose minimal public API,
  update call sites in main.py — no behavior changes at runtime.
- The `is_closing` callable boundary is the key design decision: it replaces the direct
  `self._closing` attribute read inside the polling loop, decoupling the server module
  from the app instance entirely.

</specifics>

<deferred>
## Deferred Ideas

- **Other SOLID principles** (OCP, LSP, ISP, DI) — no natural application points identified;
  not applicable to single-class desktop app at current scale.
- **Job execution decomposition** (generate + translate flows) — natural next step after
  server lifecycle; candidate for phase 999.23.1 or a follow-on refactor phase.
- **Help dialogs decomposition** (`_show_tutorial`, `_show_language_pairs`, `_show_about`) —
  complete isolation from app state; good quick-win for a future phase.
- **OS theme sync decomposition** (`sync_with_os`, `_start_os_theme_listener`) — pure
  system concern; candidate for extraction after server lifecycle.
- **Dependency injection** — deferred; revisit when testability becomes a concrete need.

</deferred>
