# Phase 999.19: Config — Configurable JSON Config File Location - Discussion Log

> **Audit trail only.** Do not use as input to planning, research, or execution agents.
> Decisions are captured in CONTEXT.md — this log preserves the alternatives considered.

**Date:** 2026-04-13

---

## Area 1: Override Mechanism

**Question:** How should users override the config file path?

**Options presented:**
1. Both CLI flag (`--config`) + env var (`GENSUBTITLES_CONFIG`) — full priority chain
2. Env var only — covers all entry points uniformly, CLI flag deferred
3. CLI flag only — CLI users only, env var deferred

**User selected:** 2 — Env var only

**Notes:** Env var approach covers CLI, API, and GUI entry points without touching argument parsing. CLI flag deferred to a future phase.

---

## Area 2: Portable Mode

**Question:** Should this phase include auto-detection of a config file next to `main.py`?

**Options presented:**
1. Include portable mode — use `gensubtitles.json` next to `main.py`
2. Include portable mode — use `settings.json` next to `main.py`
3. Defer portable mode — env var is enough for this phase

**User selected:** 3 — Defer portable mode

**Notes:** Env var mechanism is sufficient for the use cases targeted (portable installs, shared network configs, version-controlled configs).

---

## Area 3: GUI Settings Path Display

**Question:** How should the active config path be shown in the Settings panel?

**Options presented:**
1. Read-only label showing the active config path
2. Read-only label + "Open folder" button (reveals config dir in file explorer)
3. Skip GUI display entirely this phase

**User selected:** 2 — Read-only label + "Open folder" button

**Notes:** Since D-01 chose env var only (no GUI-driven path change), the display is purely informational + navigation. Reuses the same "open folder" cross-platform pattern already in the GUI.

---

## Area 4: `settings_path()` Caching

**Question:** Should `settings_path()` cache the resolved path or read the env var fresh on every call?

**Options presented:**
1. Read env var fresh on every `settings_path()` call (simple, no caching)
2. Cache at first call (module-level singleton)

**User selected:** 1 — Read env var fresh on every call

**Notes:** No mid-run path changes expected; fresh read is simpler to implement and easier to test (no need to reset module state between tests).
