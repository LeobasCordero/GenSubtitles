# Phase 999.30 — Context: REFACTOR GUI con Tabs por Funcionalidad

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Reorganizar la GUI de 2 tabs → 6 tabs (CTkTabview), donde cada tab encapsula un paso del pipeline con su propio formulario y consola de logs. El tab "Traducción de Subtítulos" (tab 2) permanece sin cambios. Los tabs 3-6 son versiones independientes de cada paso del pipeline.

</domain>

<decisions>
## Implementation Decisions

### D-01: Estructura de tabs

6 tabs en el CTkTabview:
1. **"Generar Subtítulos"** — flujo completo (Extract→Transcribe→Translate→Write en un solo botón) + log
2. **"Traducción de Subtítulos"** — sin cambios (igual que hoy, `_build_translate_tab()` intacto)
3. **"Extraer Audio"** — solo extracción: video entrada + archivo salida (.wav) + botones Extraer/Limpiar + log
4. **"Transcribir"** — solo transcripción: archivo entrada (.wav) + archivo salida (.json/.srt) + botones Transcribir/Limpiar + log
5. **"Traducción"** — solo traducción: archivo entrada + idioma origen + idioma destino + motor traducción + botones Traducir/Limpiar + log
6. **"Escribir Subtítulo"** — solo escritura SRT: archivo entrada + formato salida (srt, ssa) + botones Escribir/Limpiar + log

### D-02: Tab 1 — Layout de 2 paneles (sin stepper)

- **Sin CTkSwitch, sin stepper de 4 etapas** — el modo paso a paso queda eliminado del tab 1 (las tabs 3-6 son la interfaz paso a paso).
- Layout: **2 paneles side-by-side** (peso 1:1):
  - Panel izquierdo (`files_frame`): video entrada (Entry + Explorar) + archivo salida (Entry + Guardar como)
  - Panel derecho (`config_frame`): sección "Configuración" con 4 dropdowns (idioma origen, idioma destino, motor traducción, formato salida) + botón "Generar Subtítulos" (acento) + botón "Limpiar campos" (secundario) + CTkTextbox consola de logs
- `minsize` puede ajustarse — sin los 3 paneles, puede reducirse a ~800×500. El agente decide el valor exacto.

### D-03: Consola de logs en cada tab (tabs 1, 3, 4, 5, 6)

- Cada tab tiene su propia `CTkTextbox` (estado disabled, auto-scroll, fondo `p("input_bg")`, fuente mono).
- Errores van a la consola de logs del tab activo — no popups/messagebox.
- Misma implementación que el `_log()` helper actual: `configure(state="normal")` → insert → `configure(state="disabled")` → `see("end")`.
- Cada tab tiene su propio método `_log_<tab>(msg)` o se reutiliza un helper genérico que recibe la referencia al textbox.

### D-04: Flujo de datos entre tabs (pre-fill del siguiente paso)

- Al completar una operación en un tab, el archivo output se **pre-llena automáticamente** como input en el siguiente tab secuencial:
  - Tab 3 completa (WAV generado) → pre-llena input de Tab 4
  - Tab 4 completa (transcripción generada) → pre-llena input de Tab 5
  - Tab 5 completa (traducción generada) → pre-llena input de Tab 6
- El usuario puede modificar el campo pre-llenado manualmente.
- Tab 1 (flujo completo) no pre-llena nada en las otras tabs — es independiente.

### D-05: Working directory por tab

- Cada tab deriva su working dir del **path de su propio archivo de entrada** (misma lógica que `_get_effective_work_dir` actual: directorio del archivo + stem).
- **Fallback:** Si el input de la tab está vacío, se usa el working dir del tab 1 (video de entrada) si está disponible, o del tab 3 (audio de entrada) si el tab 1 también está vacío.
- La lógica de fallback se implementa como un método compartido `_get_tab_work_dir(input_var, fallback_vars)`.

### D-06: StringVars y callbacks existentes

- Los `StringVar`s del tab 1 (`_input_var`, `_output_var`, `_source_lang_var`, `_target_lang_var`, `_output_format_var`, `_engine_var`) se mantienen.
- Cada tab individual (3-6) tiene sus propios `StringVar`s con prefijo del tab: `_extract_input_var`, `_extract_output_var`, `_transcribe_input_var`, etc.
- El callback `_on_generate()` del tab 1 permanece sin cambios funcionales.
- Cada tab individual (3-6) llama al endpoint API correspondiente (`/steps/extract`, `/steps/transcribe`, `/steps/translate`, `/steps/write`).

### D-07: Localización

- Todos los strings visibles de los nuevos tabs deben registrarse en `locale.py` mediante `s()` — no strings hardcodeados.
- Nombres de tabs, labels, botones, placeholders → todos por el sistema `s()` / `set_language()`.

### the agent's Discretion

- Tamaño `minsize` de la ventana con 6 tabs (el agente elige el valor apropiado).
- Método de log: helper genérico vs métodos separados por tab (el agente elige según legibilidad).
- Organización interna del código: si tabs 3-6 comparten suficiente estructura, el agente puede refactorizar con un helper `_build_step_tab(tab_ref, ...)` para evitar repetición.
- Estilo de los botones "Limpiar campos" en tabs 3-6 (secundario como en tab 1, o el agente decide).

</decisions>

## Existing Code Insights

### Reusable Assets
- `_log(msg)` helper (línea ~567): implementación de log a CTkTextbox — extraer como `_log_to(textbox, msg)` para reutilizar en todos los tabs.
- `_get_effective_work_dir()` (línea ~577): lógica de working dir derivado del input — base para `_get_tab_work_dir()`.
- `_build_translate_tab()` (línea ~943): tab 2 existente, no se toca — sirve como referencia de estructura de tab.
- `apply_accent_btn_style()`, `apply_secondary_btn_style()`, `apply_entry_style()`: helpers de estilo existentes, usar en todos los tabs nuevos.
- `_on_step_extract()`, `_on_step_transcribe()`, `_on_step_translate()`, `_on_step_write()` (líneas 699-746): lógica de cada paso individual — reutilizar como callbacks en tabs 3-6.
- `_run_step_in_bg()` (línea ~662): ejecutor genérico de pasos en background — reutilizable directamente en tabs 3-6.

### Established Patterns
- CTkTabview con `add()` y `tab()` para obtener el frame del tab.
- StringVars como fuente de verdad del estado — no leer widgets directamente.
- `self.after(0, callback)` para actualizar UI desde threads de background.
- `p()` para colores, `font()` para tipografía, `s()` para strings.

### Integration Points
- `_apply_ui_language()` (línea ~2013): debe extenderse para los nuevos labels/botones de todos los tabs nuevos.
- `_apply_theme()` (línea ~1627): aplica colores a widgets — debe cubrir los nuevos frames de tabs.
- `_relabel_tabview_headers()` (línea ~1957): renombra las tabs del CTkTabview al cambiar idioma — debe incluir los 4 nuevos nombres de tab.

</code_context>

<specifics>
## Specific Ideas

- Tab 1 layout: 2 paneles (izquierdo archivos, derecho config+botones+log) — referencia directa del usuario.
- Consola de logs en todos los tabs funcionales (1, 3, 4, 5, 6) — errores en consola, no popups.
- Pre-fill automático entre tabs secuenciales al completar operación.
- Working dir con fallback en cadena: input propio → tab 1 → tab 3.
