<domain>
## Phase Boundary

Improve subtitle translation quality in two dimensions:

1. **Context-batching (Argos):** Concatenate all segment texts into a single block before passing to
   Argos Translate, so the model has cross-sentence context instead of seeing each line in isolation.
   Split the result back to per-segment texts after translation.

2. **Alt engine support:** Add DeepL Free API and LibreTranslate (self-hosted) as optional
   higher-quality translation engines, selectable at runtime via CLI flag and API param.
   Argos Translate remains the default offline engine.

**Scope:** Changes to `gensubtitles/core/translator.py`, `gensubtitles/core/settings.py`,
`gensubtitles/cli/main.py`, `gensubtitles/api/routers/subtitles.py`, and `gensubtitles/gui/main.py`.
No changes to `pipeline.py`, `transcriber.py`, `srt_writer.py`, or `audio.py`.

Implementation must be done in a **new git branch created from remote** (not from the current
local branch head). Branch already created: `feature/999.12-translation-quality-context-engine`.

</domain>

<decisions>
## Implementation Decisions

### Batched Translation â€” Delimiter Format
- **D-01:** Use numbered XML markers as delimiters: `<1>text one</1><2>text two</2>â€¦<N>textN</N>`.
  The full batch string is a concatenation of all `<N>seg.text</N>` entries with no extra separator
  between them. After translation, parse the result with a regex to extract the text content of
  each `<N>â€¦</N>` tag.

### Batched Translation â€” Mismatch Handling
- **D-02:** After splitting the translated block back by markers, if the recovered segment count
  does not equal the input segment count, raise `RuntimeError` with a message describing the
  mismatch (input count vs output count). Do **not** silently fall back to per-segment translation.

### Batched Translation â€” Pivot Chain Handling
- **D-03:** Apply batching to **every hop** in a pivot chain (e.g., `jaâ†’enâ†’es` batches the
  `jaâ†’en` hop and then batches the `enâ†’es` hop). The markers survive between hops because they
  are re-applied at each hop boundary.

### Engine Upgrade â€” Scope
- **D-04:** Both **DeepL Free API** and **LibreTranslate (self-hosted)** are in scope for this
  phase alongside the Argos batching improvement. Argos remains the default engine and the
  offline fallback (selectable explicitly or when no engine flag is given).

### Engine Selection â€” Configuration Surface
- **D-05:** Engine is selected per-run via:
  - **CLI:** `--engine argos|deepl|libretranslate` flag (default: `argos`)
  - **API:** `engine` body/query parameter (default: `"argos"`)
  - **GUI:** `CTkOptionMenu` (options: Argos / DeepL / LibreTranslate) that appears **only when
    a target language is selected** and is hidden (`.grid_remove()`) when no target language is
    chosen. Mirrors the existing target-language visibility pattern.
  - No persistent `AppSettings` default â€” engine is always specified at call time.

### Credentials â€” Storage
- **D-06:** DeepL and LibreTranslate credentials are stored in `settings.json` via two new
  `AppSettings` fields:
  - `deepl_api_key: str = ""`
  - `libretranslate_url: str = ""` (e.g., `"http://localhost:5000"`)
  - `libretranslate_api_key: str = ""` (empty string = no API key, valid for open instances)
  The existing `save_settings()` / `load_settings()` / `AppSettings` dataclass pattern handles
  persistence automatically.

### Credentials â€” Missing/Failed Behavior
- **D-07:** If the selected engine's credentials are missing (empty string) or the API call fails
  (network error, auth error, HTTP 4xx/5xx), raise immediately with a descriptive error message.
  Do **not** silently fall back to Argos.
  - Missing DeepL key: `"DeepL API key not configured. Set deepl_api_key in settings."`
  - Missing LibreTranslate URL: `"LibreTranslate URL not configured. Set libretranslate_url in settings."`
  - API failure: re-raise the underlying error with engine name in the message.

### Agent's Discretion
- Whether to introduce an `Engine` enum or string literal type for the engine parameter vs plain
  `str` with validation.
- Whether the DeepL and LibreTranslate translation functions are standalone functions in
  `translator.py` or thin classes.
- Exact HTTP client used for DeepL (the `deepl` PyPI package or raw `httpx`/`requests`).
- Exact HTTP client used for LibreTranslate (`requests` or `httpx`; LibreTranslate has a simple
  REST API at `POST /translate`).
- Whether `translate_segments()` gains an `engine: str = "argos"` parameter or whether a new
  public function (e.g., `translate_segments_with_engine()`) wraps it.
- GUI: exact grid row for the engine dropdown (below target-language row, above Generate button).
- GUI: whether the engine dropdown is disabled during pipeline execution (follow existing
  disable/enable cycle from phases 999.4+).

## Existing Code Insights

### Reusable Assets
- `TranslatedSegment` namedtuple (`start, end, text`) â€” output type unchanged; batching is purely
  internal to `translate_segments()`.
- `find_route()` / `ensure_route_installed()` â€” already handles multi-hop; batching slots in at
  the per-hop translate call, not at the route-finding level.
- `AppSettings` dataclass + `save_settings()` / `load_settings()` â€” credential fields drop in as
  new `str = ""` fields; no changes to the persistence mechanism.
- `CTkOptionMenu` + `StringVar` pattern â€” already established in GUI for target-language dropdown
  (phase 999.9); engine dropdown reuses the same pattern.

### Established Patterns
- Per-segment translate call is inside a `for hop_from, hop_to in route: for seg in current:`
  double loop in `translate_segments()`. Batching replaces the inner per-segment loop with a
  single bulk call per hop.
- CLI flags use `typer.Option` with `Optional[str]`; new `--engine` follows the same shape.
- API optional params use `Optional[str] = None` or `str = "argos"` in the route handler.
- GUI widgets hidden with `.grid_remove()` and shown with `.grid()` based on state; engine
  dropdown visibility is driven by a `trace` on `_target_lang_var` or by the existing
  `CTkOptionMenu` `command` callback.

### Integration Points
- `translate_segments()` is called from `pipeline.py` (line ~80); adding an `engine` parameter
  there threads through to CLI and API without changing the pipeline's public signature beyond
  adding an optional kwarg.
- `AppSettings` is loaded once in the GUI `__init__` and saved on settings change; new credential
  fields are read at the point of engine invocation, not at startup.

<specifics>
## Specific Ideas

- Numbered XML markers `<N>â€¦</N>` chosen for unambiguity: extremely unlikely to appear in natural
  subtitle text, and the regex `<(\d+)>(.*?)</\1>` with `re.DOTALL` reliably extracts them even
  if Argos adds whitespace around tag content.
- Batching applies to every hop so that a `jaâ†’enâ†’es` translation gets context benefit at both
  the Japaneseâ†’English and Englishâ†’Spanish steps.
- DeepL Free API has a 500k character/month free tier; LibreTranslate supports open (no-key)
  instances as well as API-key-gated deployments â€” `libretranslate_api_key` defaults to `""`
  to support both modes.
