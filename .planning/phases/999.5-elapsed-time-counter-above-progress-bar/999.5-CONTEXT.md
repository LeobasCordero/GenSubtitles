# Phase 999.5: GUI — Elapsed Time Counter Above Progress Bar - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Display a real-time elapsed time counter (e.g., `00:01:23`) above the progress bar. The counter starts at `00:00:00` when subtitle generation begins and counts up once per second. It is hidden at rest. After generation completes (success or error), the final elapsed time lingers on screen. It only resets and hides when the next generation run starts.

</domain>

<decisions>
## Implementation Decisions

### Visibility at Rest
- **D-01:** The elapsed counter is hidden when no generation is running (same pattern as the progress bar — `.grid_remove()` at build time, `.grid()` on generate start). It does NOT show `00:00:00` at rest.

### Post-Completion Behavior
- **D-02:** After `_finish_generate` is called (success or error), the counter stays visible showing the final elapsed time. It is NOT hidden in `_finish_generate`. The progress bar still disappears as before — the counter lingers independently.

### Clear Button / Next Run Reset
- **D-03:** The Clear button does NOT hide or reset the counter. The counter only resets (back to `00:00:00`) and then starts counting again at the beginning of the next generation run (in `_on_generate`). First call to `_on_generate` hides any lingering counter, resets the clock, then shows it.

### Agent's Discretion
- Exact naming of the elapsed timer after-handle attribute (e.g., `self._elapsed_timer`) and start-time attribute (e.g., `self._elapsed_start`)
- Whether to use `time.monotonic()` or `time.time()` for elapsed tracking (monotonic preferred — immune to system clock changes)
- Counter update interval (1000 ms recommended for once-per-second ticks)
- Exact grid row insertion (counter goes above progress bar — current Row 6 — so counter becomes Row 6, progress bar shifts to Row 7, stage label to Row 8)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `self._stage_timer = self.after(2500, self._advance_stage, idx + 1)` — exact pattern for a repeating after-based timer; elapsed counter uses `self.after(1000, self._tick_elapsed)` equivalent
- `self.after_cancel(self._stage_timer)` in `_finish_generate` — same cancel pattern applies to the elapsed timer
- `self._progress_bar.grid_remove()` / `self._progress_bar.grid()` — exact show/hide pattern for the counter label
- `ctk.CTkLabel(self._frame, text="")` with `.grid(row=7, ...)` — existing stage label; counter uses same widget type

### Established Patterns
- Widgets that appear only during generation: hidden with `.grid_remove()` after creation in `_build_ui`, shown with `.grid()` in `_on_generate`
- Timer handles stored as instance attributes (`self._stage_timer`); must be cancelled in `_finish_generate` and checked for `self._closing` before ticking
- `self._closing` guard in all after-callbacks to prevent updates during shutdown

### Integration Points
- `_build_ui`: create `self._elapsed_label` as a `CTkLabel`, call `.grid_remove()` immediately after `.grid()` to hide it initially. Insert at new Row 6 (shift progress bar to Row 7, stage label to Row 8).
- `_on_generate`: hide/reset counter (`self._elapsed_label.configure(text="00:00:00")`), record `self._elapsed_start = time.monotonic()`, show label (`.grid()`), start tick loop
- `_finish_generate`: cancel elapsed timer via `after_cancel`, leave label visible with final time (do NOT call `.grid_remove()`)
- `__init__`: initialize `self._elapsed_timer = None` and `self._elapsed_start = 0.0` alongside `self._stage_timer = None`

</code_context>

<specifics>
## Specific Ideas

- Counter format: zero-padded `HH:MM:SS` — `f"{h:02d}:{m:02d}:{s:02d}"` where h = elapsed//3600, m = (elapsed%3600)//60, s = elapsed%60
- The counter resets to `00:00:00` at the very start of `_on_generate` (before starting the tick), so if a lingering value is showing, it flashes to zero then starts counting

</specifics>
