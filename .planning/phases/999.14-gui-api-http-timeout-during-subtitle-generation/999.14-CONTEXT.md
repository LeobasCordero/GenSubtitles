# Phase 999.14: GUI — HTTP Timeout During Subtitle Generation - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the blocking `POST /subtitles` HTTP call with an SSE-based async job pattern so long transcription+translation runs never hit a read timeout. Add a Cancel button that lets the user abort a running job. The `/progress` polling model is retired in favour of the SSE stream.

**Not in scope:** Burn-in, streaming audio, multi-job queues, CLI/API-consumer changes beyond the GUI.

</domain>

<decisions>
## Implementation Decisions

### Fix Approach — SSE job pattern
- **D-01:** Replace the blocking `POST /subtitles` (timeout=3600) with a 3-step SSE pattern:
  1. `POST /subtitles/async` → returns `{"job_id": "<uuid>"}` immediately
  2. `GET /subtitles/{job_id}/stream` → SSE stream sending progress events until done
  3. `GET /subtitles/{job_id}/result` → returns the SRT/SSA bytes on completion
- **D-02:** The existing `GET /progress` endpoint and `_poll_progress()` GUI method are retired — the SSE stream replaces them.
- **D-03:** Implement on a new feature branch (e.g. `feature/sse-job-pattern` or similar — name at agent discretion).

### SSE Progress Events
- **D-04:** SSE events mirror the current `_progress` dict shape: `stage`, `label`, `current`, `total`. Final success event: `data: {"stage": "done", "label": "Complete", ...}`. Error event: `data: {"stage": "error", "label": "<message>"}`.

### Error UX
- **D-05:** If the SSE connection drops or the server returns an error event, show a friendly human-readable message in the existing error display area — no retry button. The user re-clicks Generate manually. Raw exception strings (e.g. `HTTPConnectionPool...`) must not be shown.

### Cancel Button
- **D-06:** Add a Cancel button visible **only during active generation**, placed inline in the progress area below the progress bar (same row as the stage label or directly below it). It is hidden when idle.
- **D-07:** Cancel mechanism: `DELETE /subtitles/{job_id}` sets a cancellation flag server-side. The pipeline checks the flag between stages (post audio extraction, post transcription, post translation). On cancellation: temp files cleaned up, SSE stream sends `{"stage": "cancelled", "label": "Cancelled"}` and closes.
- **D-08:** After cancel completes, GUI shows "Generation cancelled." in the status area and resets to idle state (same reset path as error).

### the agent's Discretion
- SSE client library choice (use `httpx` with `iter_lines` or `requests` with `stream=True` — whichever fits cleanest given existing `requests` usage in the GUI).
- Job ID format (UUID4 is fine).
- Exact placement of Cancel button within the progress area — use judgment to match existing UI style.
- Thread-safety implementation of the cancellation flag (threading.Event is idiomatic).

</decisions>

## Existing Code Insights

### Reusable Assets
- `_set_progress(stage, label, current, total)` — existing helper; can be reused to update state that SSE streams from
- `_apply_progress(data)` — existing Tk-thread method; can be reused to apply SSE event data to UI
- `_finish_generate(error, output_path)` — existing completion handler; reuse for SSE done/error events
- `_p(key)` — palette helper; use for Cancel button colours
- `_PALETTES["Dark"/"Light"]["progress_err"]` / `["accent"]` — colour tokens for Cancel button

### Established Patterns
- Deferred imports: `import requests as req` inside method bodies (not at module top)
- Background threads for HTTP calls (`threading.Thread(target=..., daemon=True).start()`)
- `self.after(0, callback)` to schedule Tk UI updates from threads
- `self._closing` guard before any `self.after()` call

### Integration Points
- `_run_api_call()` → replace with new SSE-aware method that posts to `/subtitles/async`, then opens SSE stream
- `_poll_progress()` → retire (remove entirely once SSE replaces it)
- `gensubtitles/api/routers/subtitles.py` → add `POST /subtitles/async`, `GET /subtitles/{job_id}/stream`, `GET /subtitles/{job_id}/result`, `DELETE /subtitles/{job_id}`

</code_context>

<specifics>
## Specific Ideas

- Cancel button appears inline in the progress area, visible only during active generation — not a permanent UI element.
- The 3-step SSE pattern (async POST → SSE stream → result GET) is the canonical design — not inline SSE on the POST response.

</specifics>

<deferred>
## Deferred Ideas

- Auto-retry on connection drop — user re-clicks Generate manually (out of scope per D-05)
- Multi-job queue / concurrent jobs — single-user desktop tool, one job at a time
- Cancel from CLI or API clients — GUI-only scope for this phase

</deferred>
