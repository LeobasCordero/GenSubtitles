## Area 1: Fix Approach

**Q:** How should the GUI connect to the local API for long jobs?

**Options presented:**
- A — Remove read timeout (keep connect timeout ~10s) — 1-line fix
- B — SSE streaming for progress + connection keep-alive — new API endpoints
- C — Async job queue + `/status` polling — full async refactor
- D — Better error message + retry only

**User selected:** Option B — SSE streaming

---

**Q:** SSE design — which pattern?

**Options presented:**
1. Job ID → SSE progress stream → result endpoint (clean 3-step)
2. Job ID → SSE stream with final `done` event → result endpoint
3. Inline SSE response on POST (non-standard, SSE on POST)

**User selected:** Option 1 — Job ID + progress stream + result endpoint

---

## Area 2: Error UX

**Q:** If the SSE connection drops or server crashes mid-job, what should the user see?

**Options presented:**
1. Friendly message + retry button
2. Friendly message only — clear description, user re-clicks Generate
3. Auto-retry once, then friendly message

**User selected:** Option 2 — Friendly message only

---

## Area 3: Scope / Cancel

**Q:** Should this phase include a Cancel button?

**User decision:** Yes — include Cancel button.

---

**Q:** Cancel mechanism?

**Options presented:**
1. Best-effort cancel via DELETE — flag checked between pipeline stages, temp files cleaned
2. Kill-and-restart the API server process
3. UI-only cancel — GUI stops polling, server runs to completion

**User selected:** Option 1 — Best-effort flag-based cancel via `DELETE /subtitles/{job_id}`

---

**Q:** Where should the Cancel button appear?

**Options presented:**
1. Replaces the Generate button while a job is running
2. Appears next to Generate as a separate button (always visible, disabled when idle)
3. Appears inline in the progress area below the progress bar (only visible during generation)

**User selected:** Option 3 — Inline in progress area, only visible during generation
