## Area 1: File name/location

**Question:** Where does the new palette module live?

**Options presented:**
1. `gui/palette.py` — focused name, matches phase goal exactly
2. `gui/theme.py` — broader name that could absorb fonts/typography later
3. `gui/colors.py` — explicit about colors only

**Selected:** Option 2 — `gui/theme.py`

---

## Area 2: What moves

**Question:** Just `_PALETTES` + `_p()`, or also the typography block?

**Options presented:**
1. Colors only — `_PALETTES` + `_p()` now; typography stays until 999.22
2. Colors + typography together — move both to `theme.py`; 999.22 focuses on widget configs/paddings

**Selected:** Option 2 — Colors + typography together

---

## Area 3: Public API shape

**Question:** Keep private names or make public?

**Options presented:**
1. Keep private — `_p()`, `_font()` stay underscore-prefixed in `theme.py`; `main.py` imports with underscore
2. Make public — `p()`, `font()` in `theme.py`; `main.py` calls directly without aliasing
3. Agent's discretion

**Selected:** Option 2 — Public names `p()` and `font()`

---

## Area 4: Extensibility design

**Question:** Dict-of-dicts vs TypedDict vs registry pattern?

**Options presented:**
1. Dict-of-dicts (keep as-is) — adding palette = adding a key
2. TypedDict for palette entries — type-checked keys
3. Registry pattern — `register_palette()` for plugin-style addition

**Selected:** Option 1 — Dict-of-dicts, unchanged structure
