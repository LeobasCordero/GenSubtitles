## Import Design

`theme.py` will use `import customtkinter as ctk` at module level — matches existing `main.py` pattern. No lazy imports needed.

**Import line to add to `main.py`** (after existing `import customtkinter as ctk`):
```python
from .theme import p, font
```

The `ctk` import stays in `main.py` — it's needed for OS theme detection, widget classes, and startup calls like `ctk.set_appearance_mode()`.

## Risk Assessment

- **Risk: Missing a call site** — Mitigate: grep for `_p(` and `_font(` after edit; assert zero remaining occurrences before commit.
- **Risk: Import order causing circular import** — None: `theme.py` only imports `customtkinter`, which is also a top-level import in `main.py`. No circular dependency.
- **Risk: `from __future__ import annotations`** — `main.py` has this; `theme.py` should include it too for consistency.

## Validation Architecture

This is a pure code-move refactor. Behavioral equivalence is guaranteed when:

1. `theme.py` exists and exports `p` and `font`
2. `main.py` imports `from .theme import p, font`
3. Zero occurrences of `_p(` or `_font(` remain in `main.py`
4. Existing test suite passes unchanged (no GUI tests import internal theme symbols)
