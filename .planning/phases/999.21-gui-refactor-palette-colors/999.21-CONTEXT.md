# Phase 999.21: REFACTOR — Palette Colors Separation - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract the color palette definitions and typography scale from `gensubtitles/gui/main.py`
into a new dedicated module `gensubtitles/gui/theme.py`. Update all call sites in `main.py`
to import from `theme.py`. No behavior changes — pure structural refactor.

Scoped to `gensubtitles/gui/` only. No API, pipeline, CLI, or settings changes.

Implementation must be done in a **new git branch created from remote** (branch `phase/999.21-gui-theme-module` already created from `origin/main`).

</domain>

<decisions>
## Implementation Decisions

### New Module
- **D-01:** Create `gensubtitles/gui/theme.py` as the destination for all moved code.
  Name chosen as "theme" (not "palette" or "colors") to accommodate both color and
  typography concerns under one roof, leaving room for future theme-level additions
  without another rename.

### What Moves
- **D-02:** Move the following blocks from `gui/main.py` to `gui/theme.py`:
  - **Color palettes:** `_PALETTES` dict (Dark + Light entries, ~40 lines)
  - **Color accessor:** `_p()` function → renamed to `p()` (public)
  - **Typography scale:** `_FONT_SIZES`, `_FONT_FAMILIES`, `_BOLD_ROLES`, `_font()` → `font()` (public)
  - The section header comments for both blocks move with the code.
- **D-03:** `gui/main.py` retains everything else (OS theme detection, `_STRINGS*`, widget
  classes, etc.). Only the two theme blocks identified in D-02 are removed from `main.py`.

### Public API Shape
- **D-04:** In `theme.py`, the functions are **public**: `p(key: str) -> str` and
  `font(role: str = "body") -> ctk.CTkFont`. No underscore prefix.
- **D-05:** `main.py` updates its import to `from .theme import p, font` and renames
  all ~60+ call sites from `_p(…)` → `p(…)` and `_font(…)` → `font(…)`.
  No aliasing — call sites use the public names directly.
- **D-06:** `_PALETTES`, `_FONT_SIZES`, `_FONT_FAMILIES`, `_BOLD_ROLES` remain
  private (underscore-prefixed) inside `theme.py` — they are implementation details
  of the module, not part of its public API. Only `p()` and `font()` are public.

### Extensibility
- **D-07:** Dict-of-dicts structure for `_PALETTES` is unchanged. Adding a new palette
  = adding a new key to the dict. `p()` already falls back to `"Dark"` for unknown modes.
  No registry pattern or TypedDict needed for this phase.

### Downstream Impact on 999.22
- **D-08:** Phase 999.22 (Separate GUI Styles) now focuses on widget configs and paddings
  only. Font/typography extraction is completed here in 999.21, so 999.22's scope
  is narrowed accordingly.

### Agent's Discretion
- Whether to export `_PALETTES` itself (e.g. for testing) or keep it strictly private.
- Whether `theme.py` imports `customtkinter` at module level or lazily inside `p()`/`font()`.
  (Current `main.py` uses top-level `import customtkinter as ctk` — keeping that pattern
  in `theme.py` is fine.)
- Exact order of sections within `theme.py` (colors first, then typography, or merged).

</decisions>

## Existing Code Insights

### Reusable Assets
- `_PALETTES` (lines 40–78): dict with "Dark" and "Light" keys, each mapping 16 color
  token names to hex strings. Self-contained — no dependencies on other `main.py` globals.
- `_p(key)` (lines 80–83): calls `ctk.get_appearance_mode()` and indexes `_PALETTES`.
  Only dependency is `ctk` (customtkinter) and `_PALETTES`.
- `_FONT_SIZES`, `_FONT_FAMILIES`, `_BOLD_ROLES`, `_font(role)` (lines 98–132):
  typography scale. `_font()` calls `ctk.get_appearance_mode()` and returns a
  `ctk.CTkFont`. Only dependency is `ctk` and the three dicts.

### Established Patterns
- All theme code is currently at module level in `main.py` (not inside a class).
  The new `theme.py` should follow the same pattern — module-level definitions, no class.
- `main.py` uses `import customtkinter as ctk` at the top. `theme.py` should do the same.

### Integration Points
- `main.py` → `theme.py`: single `from .theme import p, font` import replaces the
  inline definitions. All `_p(…)` and `_font(…)` call sites in `main.py` need renaming.

</code_context>
